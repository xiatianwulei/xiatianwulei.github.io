# KMP 新功能开发指南（以「系统通知设置」为例）
> 本指南以 `app-main/mine/src/commonMain/kotlin/com/app/main/mine/setting/systemNotification` 为示例，说明在现有 KMP 架构下从零开发一个功能的标准流程。

---

## 一、架构分层概览

| 模块 | 职责 | 典型内容 |
|------|------|----------|
| `kmp-core/core` | 基础设施 / MVI 基类 | `BaseViewModel`、`BaseUseCase`、`Result`、`Response`、`UIComponent` 等 |
| `kmp-core/base` | 平台上下文 | `PlatformContext`、`getPlatformContext()` |
| `kmp-common/common` | 平台能力（expect/actual） | 通知状态、拨号、图片选择、跳转链接等 |
| `app-common/domain` | 领域层 | 领域模型、Repository 接口、UseCase |
| `app-common/data` | 数据层 | API 接口/实现、DTO、Repository 实现 |
| `app-common/datasource` | 旧网络/数据源层 | 现有 Service、Response DTO（新功能尽量走 data/domain） |
| `app-common/interactor` | 旧业务用例层 | 现有 Interactor（新功能尽量走 domain UseCase） |
| `app-main/<feature>` | 表现层 | Compose Screen、ViewModel、Android Activity |
| `app-main/main` | DI 汇总 | `appModule`、`dataModule` 注册入口 |

**依赖方向（严格单向）**：

```
app-main/<feature> → app-common/domain
app-common/data → app-common/domain
app-common/data → kmp-common/common / kmp-core
```

`app-main/<feature>` 不直接依赖 `app-common/data`。

---

## 二、开发流程总览

开发一个完整功能，建议按以下顺序进行：

1. **领域层（domain）**：定义数据模型、仓库接口、UseCase。
2. **数据层（data）**：定义 API、DTO、仓库实现。
3. **平台能力层（kmp-common/common）**：如需访问系统能力（通知、电话等），定义 `expect/actual`。
4. **表现层（app-main/<feature>）**：编写 ViewModel、Screen、Android Activity。
5. **入口与 DI**：注册 Activity、Koin 依赖、模块跳转方法。
6. **编译验证**：先跑 `compileKotlinMetadata`，再跑 `compileDebugKotlinAndroid`。

## 三、每个模块建什么文件、作用是什么

以「系统通知设置」功能为例，假设功能名为 `systemNotification`，所在业务模块为 `app-main/mine`。

### 3.1 领域层：`app-common/domain/src/commonMain/kotlin/com/app/common/domain/notification/`

| 文件 | 作用 |
|------|------|
| `NotificationState.kt` | 领域模型（纯数据类），对应页面需要展示/操作的业务数据。 |
| `repository/NotificationSettingRepository.kt` | 仓库接口，定义获取/保存数据的抽象契约。返回 `com.kmp.core.Result<T>`。 |
| `usecase/GetNotificationSettingUseCase.kt` | 获取通知设置的 UseCase，继承 `BaseUseCase<Unit, NotificationState>`。 |
| `usecase/SaveNotificationSettingUseCase.kt` | 保存通知设置的 UseCase，继承 `BaseUseCase<SaveNotificationSettingParam, Unit>`。 |
| `usecase/SaveNotificationSettingParam.kt` | UseCase 参数数据类。 |

**示例：NotificationState.kt**

```kotlin
package com.app.common.domain.notification

import androidx.compose.runtime.Stable

@Stable
data class NotificationState(
    val systemNoticeFlag: Boolean = false,
    val ticketReminderFlag: Boolean = false,
    // ...
)
```

**示例：Repository 接口**

```kotlin
interface NotificationSettingRepository {
    suspend fun getNotificationSetting(): Result<NotificationState>
    suspend fun saveNotificationSetting(item: String, open: Boolean): Result<Unit>
}
```

**示例：UseCase**

```kotlin
class GetNotificationSettingUseCase(
    private val repository: NotificationSettingRepository,
) : BaseUseCase<Unit, NotificationState>() {
    override suspend fun execute(params: Unit): Result<NotificationState> {
        return repository.getNotificationSetting()
    }
}
```

---

### 3.2 数据层：`app-common/data/src/commonMain/kotlin/com/app/common/data/notification/`

| 文件 | 作用 |
|------|------|
| `api/NotificationSettingApi.kt` | 网络 API 接口 + 内部实现（`NotificationSettingApiImpl`）。 |
| `dto/NotificationSettingDto.kt` | 请求/响应 DTO（`@Serializable`），以及 `toDomain()` 映射扩展。 |
| `repository/NotificationSettingRepositoryImpl.kt` | 仓库接口实现，调用 API，处理 `safeApiCall` 与 `converterResult`。 |

**示例：API 接口**

```kotlin
interface NotificationSettingApi {
    companion object {
        const val GET_NOTIFICATION_SETTING = "/member_settings/push_switch_list.api"
        const val SAVE_NOTIFICATION_SETTING = "/member_settings/save_push_switch.api"
    }

    suspend fun getNotificationSetting(): Response<NotificationStateDTO>
    suspend fun saveNotificationSetting(params: NotificationSaveRequestDTO): Response<NotificationSaveResultDTO>
}

internal class NotificationSettingApiImpl(
    private val httpClient: HttpClient,
    private val apiHost: Host,
) : NotificationSettingApi { /* ... */ }
```

**示例：Repository 实现**

```kotlin
class NotificationSettingRepositoryImpl(
    private val api: NotificationSettingApi,
) : NotificationSettingRepository {
    override suspend fun getNotificationSetting(): Result<NotificationState> {
        return safeApiCall {
            api.getNotificationSetting().converterResult {
                when {
                    this == null -> Result.Success(NotificationState())
                    else -> Result.Success(toDomain())
                }
            }
        }
    }
}
```

### 3.3 平台能力层：`kmp-common/common/src/.../kotlin/com/kmp/common/notification/`

如果功能需要访问平台原生能力（如通知状态、打电话、相册），在 `kmp-common/common` 下新建功能目录，使用 `expect/actual object`。

| 文件 | 作用 |
|------|------|
| `commonMain/.../notification/SystemNotificationSettings.kt` | `expect object SystemNotificationSettings`，声明平台无关接口。 |
| `androidMain/.../notification/SystemNotificationSettings.android.kt` | `actual object SystemNotificationSettings`，Android 实现。 |
| `iosMain/.../notification/SystemNotificationSettings.ios.kt` | `actual object SystemNotificationSettings`，iOS 实现。 |

**示例**

```kotlin
// commonMain
expect object SystemNotificationSettings {
    fun isEnabled(): Boolean
    fun openSettings()
}

// androidMain
actual object SystemNotificationSettings {
    actual fun isEnabled(): Boolean { /* 读取 NotificationManager */ }
    actual fun openSettings() { /* 跳转系统设置 */ }
}

// iosMain
actual object SystemNotificationSettings {
    actual fun isEnabled(): Boolean = false // 占位
    actual fun openSettings() = Unit
}
```

> 注意：`expect/actual object` 由编译器自动解析，**不需要在 Koin 中注册**。

### 3.4 表现层：`app-main/mine/src/commonMain/kotlin/com/app/main/mine/setting/systemNotification/`

| 文件 | 作用 |
|------|------|
| `viewmodel/SystemNotificationViewModel.kt` | MVI 核心：定义 `State`、`Event`、`Action`，继承 `BaseViewModel`。 |
| `SystemNotificationScreen.kt` | Compose 页面，接收 `ViewModel`，渲染 UI，响应生命周期。 |
| `androidMain/.../systemNotification/SystemNotificationActivity.kt` | Android 入口 Activity，调用 `setContent { SystemNotificationScreen(...) }`。 |

**ViewModel 结构**

```kotlin
// State
@Immutable
data class SystemNotificationUiState(
    val systemEnabled: Boolean = true,
    val notificationState: NotificationState = NotificationState(),
    val progressBarState: ProgressBarState = ProgressBarState.Idle,
) : ViewState

// Event
@Immutable
sealed class SystemNotificationEvent : ViewEvent {
    data object Load : SystemNotificationEvent()
    data class Toggle(val item: NotificationItem) : SystemNotificationEvent()
}

// Action
@Immutable
sealed class SystemNotificationAction : ViewSingleAction {
    data object OpenSystemSettings : SystemNotificationAction()
}

// ViewModel
class SystemNotificationViewModel(
    private val getUseCase: GetNotificationSettingUseCase,
    private val saveUseCase: SaveNotificationSettingUseCase,
) : BaseViewModel<SystemNotificationEvent, SystemNotificationUiState, SystemNotificationAction>()
```

**Screen 结构**

```kotlin
@Composable
fun SystemNotificationScreen(
    viewModel: SystemNotificationViewModel = koinInject(),
    state: State<SystemNotificationUiState> = viewModel.collectAsState(),
    errors: Flow<UIComponent> = viewModel.errors,
    events: (SystemNotificationEvent) -> Unit = viewModel.rememberEvents(),
    onBack: () -> Unit = {}

    viewModel.LaunchActions { action -> /* 处理一次性动作 */ }
    ObserveLifecycle(onResume = { events(SystemNotificationEvent.RefreshSystemStatus) })
    DisposableEffect(Unit) {
        events(SystemNotificationEvent.Load)
        onDispose { events(SystemNotificationEvent.Clear) }
    }

    DefaultScreenUI(
        errors = errors,
        progressBarState = state.value.progressBarState,
        topBar = { /* TopAppBar */ }
    ) { innerPadding ->
        // 页面内容
    }
}
```

### 3.5 DI 注册

#### 1) `app-common/data/src/commonMain/kotlin/com/app/common/data/DataModule.kt`

```kotlin
single<NotificationSettingRepository> { NotificationSettingRepositoryImpl(get()) }
```

```kotlin

single { SystemNotificationViewModel(get(), get(), get()) }
```

## 四、关键约定与注意事项

### 4.1 MVI 约定

- `State` 使用 `@Immutable` 数据类，必须实现 `ViewState`。
- `Event` 使用 `@Immutable` 密封类，必须实现 `ViewEvent`。
- `Action` 使用 `@Immutable` 密封类，必须实现 `ViewSingleAction`。
- `ViewModel` 继承 `BaseViewModel<Event, State, Action>`，在 `onTriggerEvent` 中分发事件。
- 状态更新统一使用 `update { copy(...) }`，`update` 的 lambda receiver 就是当前 State，不要写 `update { current -> copy(...) }`。

### 4.2 UseCase 约定

- 继承 `BaseUseCase<Params, ResultType>`。
- `execute(params)` 返回 `Result<T>`。
- 无参数时用 `Unit`。

### 4.3 Repository 约定

- 接口放在 `app-common/domain/<feature>/repository/`。
- 实现放在 `app-common/data/<feature>/repository/`。
- 网络异常统一用 `safeApiCall` 包裹，业务码通过 `converterResult` 处理。

### 4.4 DTO 约定

- 请求/响应 DTO 全部加 `@Serializable`。
- 服务端字段用 `@SerialName` 映射。
- 扩展函数 `fun XxxDTO.toDomain(): DomainModel` 写在同文件或单独 mapper 文件中。

### 4.5 平台能力约定

- 优先使用 `expect/actual object`。
- 放置在 `kmp-common/common/src/<sourceSet>/kotlin/com/kmp/common/<feature>/`。
- Android 实现中需要 `Context` 时，通过 `com.kmp.core.base.getPlatformContext()` 获取。

### 4.6 UI 约定

- 页面统一用 `DefaultScreenUI` 包裹，自动处理错误、Loading、TopBar。
- ViewModel 通过 `koinInject()` 注入。
- 状态通过 `viewModel.collectAsState()` 收集。
- 事件通过 `viewModel.rememberEvents()` 派发。
- 一次性动作通过 `viewModel.LaunchActions { ... }` 处理。
- 生命周期通过 `ObserveLifecycle` 监听。

## 五、最小开发清单

开发新功能时，按以下清单检查：

- [ ] `app-common/domain/<feature>/` 下创建领域模型、Repository 接口、UseCase。
- [ ] `app-common/data/<feature>/` 下创建 API、DTO、Repository 实现。
- [ ] 如需平台能力，`kmp-common/common/src/<sourceSet>/.../<feature>/` 下创建 `expect/actual`。
- [ ] `app-main/<module>/.../<feature>/viewmodel/` 下创建 ViewModel。
- [ ] `app-main/<module>/.../<feature>/` 下创建 Screen。
- [ ] `app-main/<module>/src/androidMain/.../<feature>/` 下创建 Activity。
- [ ] `AndroidManifest.xml` 注册 Activity。
- [ ] `MineModule.kt`（或对应模块入口文件）添加跳转方法。
- [ ] `DataModule.kt` 注册 Repository / API / UseCase。
- [ ] `Koin.kt` 注册 ViewModel。
- [ ] 编译验证：`compileKotlinMetadata` → `compileDebugKotlinAndroid`。
